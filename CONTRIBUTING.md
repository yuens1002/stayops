# Contributing to stayops

This started as a personal tool for managing the author's own short-term rental properties, shared publicly in case it's useful to others. The priority is self-consumption, so expect the roadmap to be driven by the author's own needs first.

## Reporting issues

Open a GitHub issue — bugs, questions, or feature ideas are all welcome. For features, note that they'll be weighed against personal usefulness first.

## Development process

This repo uses the `/commit` workflow (conventional commit → PR → review → merge) for changes. For small fixes (docs, typos, config), a normal PR is fine.

1. Fork the repo, create a branch off `main`.
2. Make your change.
3. Open a PR against `main` with a clear description of what changed and why.
4. CI must pass; a maintainer will review.

## Commit style

Conventional commits (`feat:`, `fix:`, `docs:`, `chore:`, etc.).
